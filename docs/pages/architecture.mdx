# Architecture

## Overview

Guarantee is a TEE attestation SDK for Rust web services. It provides a thin
layer on top of Intel SGX via the Gramine framework, giving every HTTP response
a cryptographic proof of origin.

The architecture has three components that compose cleanly:

1. **Key hierarchy** — a tree of keys derived inside the enclave at startup
2. **Attestation flow** — how trust is established from hardware to response
3. **TEE state model** — how application state is scoped and persisted

---

## TEE State Model

State in Guarantee is divided into three tiers with different lifetime and
sharing characteristics.

```
┌──────────────────────────────────────────────────────────┐
│                     Enclave boundary                      │
│                                                           │
│  ┌─────────────────┐   ┌─────────────────┐               │
│  │  MRENCLAVE tier │   │  MRSIGNER tier  │               │
│  │                 │   │                 │               │
│  │  Resets on      │   │  Persists across│               │
│  │  every redeploy │   │  redeploys      │               │
│  │                 │   │                 │               │
│  │  Session data   │   │  Master keys    │               │
│  │  Counters       │   │  Long-lived cfg │               │
│  │  Caches         │   │  User secrets   │               │
│  └─────────────────┘   └─────────────────┘               │
│                                                           │
│  ┌──────────────────────────────────────────────────┐    │
│  │              External tier                        │    │
│  │                                                   │    │
│  │  AES-256-GCM encrypted before leaving enclave     │    │
│  │  Safe to store in PostgreSQL / Redis / S3         │    │
│  │  Keys never visible outside the enclave           │    │
│  └──────────────────────────────────────────────────┘    │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

### MRENCLAVE tier

Sealed using the enclave's binary measurement. Only the exact same compiled
binary can unseal this data. Every redeploy produces a new MRENCLAVE and
resets this state to its `Default` value.

Suitable for: per-session state, request counters, ephemeral caches.

### MRSIGNER tier

Sealed using the hash of the enclave signing key. Any binary signed with the
same key can unseal this state, so it persists across rolling deployments and
upgrades. Only a signing key change resets it.

Suitable for: master encryption keys, long-lived configuration, user-owned
secrets that must survive redeploys.

### External tier

Data in the external tier is encrypted with AES-256-GCM before it leaves the
enclave. Encryption keys are derived from the MRSIGNER master key via
HKDF-SHA256. The plaintext never appears in external storage.

Suitable for: user records in PostgreSQL, cached tokens in Redis, audit logs
in object storage.

## Key Hierarchy

```
Hardware entropy (OsRng)
│
├─→ signing_key  (Ed25519, MRENCLAVE-scoped, hidden)
│     └─→ per-response X-TEE-Attestation signatures
│
└─→ master_key  (256-bit AES, MRSIGNER-scoped, hidden)
      │
      ├─→ derive_key(master, b"TypeName::field")  → AES-256-GCM encryption key
      │     └─→ enc:v1:k<N>:<nonce>:<ciphertext>  (stored externally)
      │
      └─→ retired_keys  (previous master keys kept for decryption fallback)
```

- **`signing_key`** is generated fresh on every enclave startup. It is bound to
  the enclave via the SGX startup quote. Per-response signatures are Ed25519
  over `SHA-256(body || timestamp_ms || request_id)`.

- **`master_key`** persists in MRSIGNER-sealed storage. Sub-keys for each struct
  type are derived deterministically from the master key using HKDF-SHA256 with
  a type-specific purpose string. Rotating the master key appends the old key to
  the `retired_keys` list so existing ciphertext can still be decrypted.

- Neither key is ever serialized in plaintext. In enclave mode, all sealed files
  are protected by Gramine's [Protected Files] feature, which applies SGX sealing
  at the OS layer.

[Protected Files]: https://gramine.readthedocs.io/en/stable/manifest-syntax.html#protected-files

## Attestation Flow

### Startup

```
1. Generate ephemeral Ed25519 signing keypair
2. Compute SHA-256(public_key) → 32 bytes of user report data
3. Write 64 bytes (key hash + zero padding) to /dev/attestation/user_report_data
4. Read DCAP quote from /dev/attestation/quote
   └─→ SGX hardware signs: MRENCLAVE, MRSIGNER, user_report_data
5. Serve quote at GET /.well-known/tee-attestation
```

### Per-request

```
1. Handler runs, produces response body
2. #[attest] macro calls state.sign_response(body, request_id)
3. payload_hash = SHA-256(body || timestamp_ms_be || request_id)
4. signature = Ed25519_sign(signing_key, payload_hash)
5. X-TEE-Attestation: v=1; sig=<b64>; hash=<hex>; ts=<ms>; key=<hex>
```

### Client verification

```
1. Fetch GET /.well-known/tee-attestation
   └─→ record public_key, mr_enclave, raw_quote
2. Verify raw_quote with Intel DCAP or a third-party verifier
3. Confirm quote.report_data[0..32] == SHA-256(public_key)
4. Receive a response with X-TEE-Attestation header
5. Recompute payload_hash = SHA-256(body || ts || request_id)
6. Verify Ed25519 signature against payload_hash using public_key
```

## Gramine Integration

In production, Guarantee runs inside a [Gramine] SGX enclave. Gramine acts as
a LibOS: it runs the Rust binary inside the SGX enclave and virtualizes OS
interfaces such as the filesystem and network.

Key manifest requirements:

```toml
# Expose the attestation pseudo-device (required)
fs.mounts = [
  { path = "/dev/attestation", type = "pseudo" },
]

# Seal MRENCLAVE state
sgx.protected_files = [
  { path = "/sealed/enclave.bin" },
]

# Seal MRSIGNER state
sgx.protected_mrsigner_files = [
  { path = "/sealed/signer.bin" },
]
```

Without the `/dev/attestation` mount, `EnclaveAttestor::initialize` will
return `SdkError::AttestationUnavailable` when `GUARANTEE_ENCLAVE=1` is set.

[Gramine]: https://gramine.readthedocs.io/

## RA-TLS (`ra-tls` feature)

When the `ra-tls` feature is enabled, Guarantee can serve HTTPS with the SGX
quote embedded directly in the TLS certificate as an X.509 extension
(OID `1.2.840.113741.1337.6`). This allows peer enclaves to verify attestation
during the TLS handshake with no additional round trips.

```
Client (peer enclave)                Server (this enclave)
│                                         │
│── TLS ClientHello ──────────────────→   │
│                                         │
│   ← TLS Certificate (X.509) ────────   │
│     extension: SGX quote                │
│       quote.report_data == SHA-256(TLS public key)
│       quote.MRENCLAVE == expected       │
│                                         │
│── Verify quote ────────────────────→   │
│── TLS Finished ────────────────────→   │
│                                         │
│ (connection now mutually attested)      │
```

See [RA-TLS](/sdk/ra-tls) for usage.
