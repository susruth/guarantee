# Field-Level Encryption

## Overview

Guarantee's field-level encryption lets you store sensitive data in external
databases or caches without ever exposing the plaintext outside the enclave.
Encryption keys are derived inside the enclave from the MRSIGNER master key and
never written to disk in plaintext.

The encrypted field format is self-describing:

```
enc:v1:<nonce_hex>:<ciphertext_hex>           ← unversioned (legacy)
enc:v1:k<version>:<nonce_hex>:<ciphertext_hex> ← versioned (recommended)
```

---

## Usage

### 1. Derive `Encrypted` on your struct

```rust
use guarantee::Encrypted;
use serde::{Deserialize, Serialize};

#[derive(Encrypted, Serialize, Deserialize, Clone, Debug)]
struct UserRecord {
    pub user_id: String,    // not encrypted — queryable in DB
    #[encrypt]
    pub email: String,      // encrypted before leaving the enclave
    #[encrypt]
    pub api_key: String,    // encrypted before leaving the enclave
}
```

The macro generates `UserRecordEncrypted` with the same fields. Encrypted
fields contain the `enc:v1:...` string; plaintext fields are copied verbatim.

### 2. Mark as `#[external]` in `state!`

```rust
use guarantee::state;

state! {
    #[mrsigner]
    MasterConfig,   // holds the master encryption key

    #[external]
    UserRecord,     // generates encrypt/decrypt methods on TeeState
}
```

### 3. Encrypt before storing, decrypt after loading

```rust
// In an axum handler:
let state = state.read().await;

// Encrypt before writing to PostgreSQL
let record = UserRecord {
    user_id: "usr_123".into(),
    email: "alice@example.com".into(),
    api_key: "sk-super-secret".into(),
};
let encrypted: UserRecordEncrypted = state.encrypt_user_record(&record)?;
db.insert(&encrypted).await?;

// Decrypt after reading from PostgreSQL
let row: UserRecordEncrypted = db.find_by_id("usr_123").await?;
let user: UserRecord = state.decrypt_user_record(&row)?;
println!("{}", user.email); // "alice@example.com"
```

## PostgreSQL Example

Store encrypted records using `sqlx`:

```rust
use sqlx::PgPool;

async fn save_user(
    pool: &PgPool,
    state: &TeeState,
    user: &UserRecord,
) -> Result<(), Box<dyn std::error::Error>> {
    let enc = state.encrypt_user_record(user)?;

    sqlx::query!(
        "INSERT INTO users (user_id, email, api_key) VALUES ($1, $2, $3)",
        enc.user_id,
        enc.email,   // "enc:v1:k1:..."
        enc.api_key, // "enc:v1:k1:..."
    )
    .execute(pool)
    .await?;

    Ok(())
}

async fn load_user(
    pool: &PgPool,
    state: &TeeState,
    user_id: &str,
) -> Result<UserRecord, Box<dyn std::error::Error>> {
    let row = sqlx::query_as!(
        UserRecordEncrypted,
        "SELECT user_id, email, api_key FROM users WHERE user_id = $1",
        user_id,
    )
    .fetch_one(pool)
    .await?;

    Ok(state.decrypt_user_record(&row)?)
}
```

## Key Derivation

Each struct type gets its own AES-256-GCM key derived from the MRSIGNER master
key using HKDF-SHA256. The derivation is deterministic: the same
`(master_key, purpose)` pair always produces the same sub-key.

```rust
use guarantee::crypto::derive_key;

let master: [u8; 32] = /* loaded from MRSIGNER sealed storage */;

// Different types get different keys
let user_key  = derive_key(&master, b"UserRecord");
let audit_key = derive_key(&master, b"AuditEntry");
// user_key != audit_key
```

Compromise of one derived key does not affect others.

## Manual Encryption

For simple string fields without the `#[derive(Encrypted)]` workflow:

```rust
use guarantee::crypto::{encrypt_field, decrypt_field};

let key = [42u8; 32]; // 256-bit key

let ciphertext = encrypt_field("my secret", &key)?;
// → "enc:v1:aabbcc...:ddeeff..."

let plaintext = decrypt_field(&ciphertext, &key)?;
// → "my secret"
```

For key rotation support, use the versioned variants:

```rust
use guarantee::crypto::{encrypt_field_versioned, decrypt_field_versioned, RetiredKeyEntry};

let encrypted = encrypt_field_versioned("secret", &key, /*version=*/1, b"purpose")?;
// → "enc:v1:k1:aabbcc...:ddeeff..."

let plain = decrypt_field_versioned(&encrypted, &key, 1, &[], b"purpose")?;
// → "secret"
```

See [Key Rotation](./key-rotation) for how to handle key rotation in
production.

## Constraints

- Only `String` fields can be marked `#[encrypt]`. Serialize other types to
  `String` (e.g., with `serde_json::to_string`) before encrypting.
- Encrypted fields are longer than their plaintext: each adds ~24 bytes of
  hex-encoded nonce plus a 16-byte GCM authentication tag plus the `enc:v1:`
  prefix. Size your database columns accordingly (e.g., `VARCHAR(512)`).
- Each encryption call generates a fresh random nonce, so encrypting the same
  value twice produces different ciphertexts. Encrypted fields are not
  searchable; use plaintext fields for query columns.
