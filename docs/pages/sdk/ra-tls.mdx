# RA-TLS

Remote Attestation TLS (RA-TLS) embeds SGX attestation quotes directly into X.509 certificates, enabling TLS connections where the certificate itself proves the server is running in a genuine enclave.

:::info
RA-TLS requires the `ra-tls` feature flag:

```toml
[dependencies]
guarantee = { version = "0.1.2", features = ["ra-tls"] }
```
:::

## `serve_ra_tls`

Start an HTTPS server with RA-TLS certificates:

```rust
use guarantee::ra_tls::server::serve_ra_tls;

// Starts both HTTP (8080) and HTTPS (8443) servers
serve_ra_tls(app, 8080, 8443).await?;
```

The RA-TLS server:
1. Generates a self-signed X.509 certificate
2. Embeds the SGX attestation quote as a certificate extension
3. Uses ECDSA P-256 for the TLS certificate key
4. Serves your axum app over both HTTP and HTTPS

## How it works

In traditional TLS, a certificate authority vouches for the server's identity. In RA-TLS, the SGX hardware itself vouches — the attestation quote embedded in the certificate proves:

- The server is running inside a genuine SGX enclave
- The enclave has a specific measurement (MRENCLAVE)
- The TLS certificate key belongs to that enclave

Clients verify the RA-TLS certificate by:
1. Extracting the embedded SGX quote from the X.509 extension
2. Verifying the quote with Intel's DCAP attestation service
3. Confirming the certificate's public key matches the quote's user report data

## When to use RA-TLS

Use RA-TLS when you need **transport-level attestation** — proving the TLS connection terminates inside a genuine enclave. This is complementary to per-response attestation:

| Approach | Proves |
|----------|--------|
| `#[attest]` (response headers) | Each response came from the attested enclave |
| RA-TLS (certificate) | The TLS connection terminates in the attested enclave |

For maximum security, use both: RA-TLS for transport attestation, `#[attest]` for data attestation.

## Inter-Enclave Connections (`connect`)

Use `guarantee::connect()` to call other enclaves with RA-TLS verification. The TLS handshake verifies the target's attestation quote automatically.

```rust
use guarantee::connect;
use guarantee::MrEnclave;

let expected = MrEnclave::new(/* known measurement */);

let conn = connect("https://oracle.internal:8443")
    .with_mrenclave(expected)
    .build()?;

let resp = conn.get("/price/BTC").await?;
let price: PriceResponse = resp.json().await?;
```

The builder supports:

| Method | Description |
|--------|-------------|
| `with_mrenclave(mr)` | Pin to a specific MRENCLAVE — rejects mismatches during handshake |
| `allow_dev_mode(bool)` | Allow connections to non-enclave targets (defaults to `true` in dev mode) |
| `build()` | Create the `EnclaveConnection` with a configured `reqwest::Client` |

`EnclaveConnection` provides `get(path)` and `post(path, body)` methods.

## WebSocket over RA-TLS

WebSocket connections served via `serve_ra_tls` are automatically attested at the TLS layer. No per-message signing is needed — the entire connection is attested during the handshake.

```rust
use guarantee::ra_tls::websocket::{WsMessage, WsSocket, WsUpgrade};
use axum::response::IntoResponse;

async fn ws_handler(ws: WsUpgrade) -> impl IntoResponse {
    ws.on_upgrade(handle_socket)
}

async fn handle_socket(mut socket: WsSocket) {
    while let Some(Ok(msg)) = socket.recv().await {
        if let axum::extract::ws::Message::Text(text) = msg {
            socket.send(axum::extract::ws::Message::Text(format!("echo: {text}"))).await.ok();
        }
    }
}
```

Add WebSocket routes to your RA-TLS router like any axum route:

```rust
let app = Router::new()
    .route("/ws", get(ws_handler))
    .route("/api", get(api_handler));

serve_ra_tls(app, 8080, 8443).await?;
```

## Dependencies

The `ra-tls` feature adds:
- `rustls` — TLS implementation
- `rcgen` — Self-signed certificate generation
- `x509-parser` — X.509 parsing
- `axum-server` — TLS-capable axum server
- `rustls-pemfile` — PEM encoding
- `time` — Certificate validity periods
- `p256` — ECDSA P-256 for certificate keys
