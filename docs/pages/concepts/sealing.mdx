# Sealing

Sealing is the process of encrypting state so it can be persisted to disk and only decrypted by the same enclave (or the same signer). Guarantee handles sealing automatically through the `state!` macro.

## Seal modes

### `SealMode::MrEnclave`

State is bound to the specific enclave binary. If the binary changes (redeployment with new code), the state cannot be unsealed. Use for ephemeral per-deployment data.

**Enclave mode**: Gramine Protected Files handle the encryption transparently — the file is encrypted with a key derived from MRENCLAVE.

**Dev mode**: Simple file I/O with a mode header. Not cryptographically secure, but preserves the same API.

### `SealMode::MrSigner`

State is bound to the signing key. As long as you sign enclaves with the same key, the state persists across binary changes. Use for long-lived secrets.

**Enclave mode**: Gramine Protected Files with MRSIGNER-derived key.

**Dev mode**: Simple file I/O with a mode header.

## How `state!` uses sealing

When you call `TeeState::initialize(seal_dir)`:

1. Attempts to unseal `seal_dir/enclave.sealed` (MRENCLAVE state)
2. Attempts to unseal `seal_dir/signer.sealed` (MRSIGNER state)
3. If either file doesn't exist, creates default state
4. Returns the hydrated `TeeState`

When you call `state.seal(seal_dir)`:

1. Serializes MRENCLAVE state to JSON
2. Seals to `seal_dir/enclave.sealed`
3. Serializes MRSIGNER state to JSON
4. Seals to `seal_dir/signer.sealed`

## Low-level sealing API

For custom use cases, the sealing functions are available directly:

```rust
use guarantee::seal::{seal_to_file, unseal_from_file, SealMode};
use std::path::Path;

// Seal arbitrary bytes
let data = b"my secret data";
seal_to_file(data, Path::new("./my-data.sealed"), SealMode::MrSigner)?;

// Unseal
let recovered = unseal_from_file(Path::new("./my-data.sealed"), SealMode::MrSigner)?;
assert_eq!(recovered, data);
```

## Signing key serialization

For cases where you need to persist or transmit Ed25519 signing keys (e.g., as part of sealed state), Guarantee provides a serde module:

```rust
use guarantee::seal::signing_key_serde;

#[derive(Serialize, Deserialize)]
struct MyState {
    #[serde(with = "signing_key_serde")]
    key: ed25519_dalek::SigningKey,
}
```

## File layout

After initialization, the seal directory looks like:

```
./sealed/
├── enclave.sealed   # MRENCLAVE-bound state (resets on redeploy)
└── signer.sealed    # MRSIGNER-bound state (persists across redeploys)
```

:::warning
In dev mode, sealed files are **not encrypted**. They contain JSON with a mode header. Never use dev mode sealed files in production.
:::
