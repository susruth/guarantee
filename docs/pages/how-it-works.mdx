# How It Works

Guarantee creates a cryptographic chain of trust from Intel SGX hardware all the way to individual HTTP responses.

## The attestation chain

```
Intel SGX Hardware
       │
       ▼
┌─────────────────────────┐
│  Enclave Measurement    │  MRENCLAVE: SHA-256 of enclave binary
│  (MRENCLAVE + MRSIGNER) │  MRSIGNER: hash of the signing key
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│    Startup Quote        │  SGX DCAP quote binding the Ed25519
│                         │  public key to the enclave identity
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│  Per-Response Signature │  Ed25519(SHA-256(body ‖ ts ‖ req_id))
│  X-TEE-Attestation      │
└─────────────────────────┘
```

## Step by step

### 1. Enclave startup

When your service starts inside a Gramine SGX enclave, Guarantee:

1. **Generates an ephemeral Ed25519 keypair** — the private key exists only in enclave memory, never serialized
2. **Writes the public key hash** to `/dev/attestation/user_report_data` (64 bytes)
3. **Reads the DCAP quote** from `/dev/attestation/quote`

The quote is a signed statement from the CPU that says: *"An enclave with measurement X, signed by key Y, reported data Z."* The reported data contains your public key hash.

### 2. Attestation endpoint

The startup quote and public key are served at:

```
GET /.well-known/tee-attestation
```

Any caller can fetch this once and cache it. The quote proves:
- The enclave is running genuine SGX hardware
- The enclave binary has a specific measurement (MRENCLAVE)
- The Ed25519 public key belongs to that enclave

### 3. Per-response signing

Every handler annotated with `#[attest]` automatically:

1. Executes your handler logic
2. Computes `payload_hash = SHA-256(response_body ‖ timestamp_ms ‖ request_id)`
3. Signs: `signature = Ed25519Sign(signing_key, payload_hash)`
4. Attaches the header:

```http
X-TEE-Attestation: v=1; sig=<base64>; hash=<hex>; ts=<unix_ms>; key=<hex(pubkey)>
```

### 4. Caller verification

A caller verifies a response in two steps:

**Once** (cacheable): Verify the startup quote with Intel's DCAP attestation service. This confirms the public key belongs to a genuine enclave.

**Per response**:
1. Parse the `X-TEE-Attestation` header
2. Recompute `hash' = SHA-256(body ‖ ts ‖ request_id)`
3. Verify `Ed25519Verify(public_key, hash', signature)`
4. Confirm the public key matches the one in the startup quote

If all checks pass, the caller has cryptographic proof that the response was produced by attested enclave code — not a compromised server, not a man-in-the-middle, not a tampered binary.

## Attestation modes

### `EveryResponse` (default)

Every handler response is signed. Use this for high-security APIs where each data point must be independently verifiable (price oracles, custody services, compliance APIs).

### `StartupOnly`

Only the startup quote is produced. Per-response signatures are skipped (the header contains empty values). Use this for services where the startup attestation is sufficient — the caller trusts that if the enclave started correctly, all responses are trustworthy.

Set via environment variable:

```bash
GUARANTEE_ATTEST_MODE=startup-only
```

Or programmatically:

```rust
let attestor = EnclaveAttestor::initialize_with_mode(
    AttestationMode::StartupOnly
).await?;
```
